## Часть 0. Что вообще такое «коллекции» в Java?

Представь, что тебе нужно хранить не одно число, а много. Массив (`int[] arr = new int[10]`) — это самый простой способ, но у него **фиксированный размер**. Добавить 11-й элемент — нельзя.

**Коллекции** — это «умные контейнеры», которые:
- сами меняют размер при добавлении/удалении;
- имеют готовые методы: добавить, удалить, найти, отсортировать;
- бывают разных видов под разные задачи.

### Иерархия коллекций (что от чего наследуется):

```
                    Collection (интерфейс)
                   /                    \
              List                      Set
           (с порядком,              (без дубликатов)
            допускает дубликаты)         |
           /        \                /       \
      ArrayList   Vector        HashSet    TreeSet
                  (наш тип1)              (наш тип2)
```

**List** — элементы идут по порядку, есть индексы (0, 1, 2...), дубликаты разрешены.
**Set** — дубликатов нет, порядок не гарантирован (кроме TreeSet — там сортировка).

## Часть 1. Что такое Vector?

`Vector` — это динамический массив. Похож на `ArrayList`, но все методы **синхронизированы** (потокобезопасны). На практике это значит:

```java
Vector<Character> v = new Vector<>();  // создали пустой вектор
v.add('a');          // добавить элемент в конец      → [a]
v.add('b');          //                                → [a, b]
v.add('c');          //                                → [a, b, c]
v.get(0);            // получить по индексу            → 'a'
v.set(1, 'X');       // заменить элемент по индексу    → [a, X, c]
v.remove(0);         // удалить по индексу             → [X, c]
v.size();            // количество элементов           → 2
v.addAll(другойВектор); // добавить все элементы из другого
```

### Почему `Vector<Character>`, а не `Vector<char>`?

В Java **дженерики** (то что в `< >`) работают только с **объектами**, а `char` — примитив (как `int`, `double`). Для каждого примитива есть класс-обёртка:

| Примитив | Обёртка |
|----------|---------|
| `char`   | `Character` |
| `int`    | `Integer` |
| `double` | `Double` |

Java **автоматически** конвертирует между ними — это называется **autoboxing/unboxing**:
```java
v.add('a');           // 'a' (char) автоматически → Character.valueOf('a')
char c = v.get(0);    // Character автоматически → char
```

## Часть 2. Что такое TreeSet?

`TreeSet` — это множество (Set), которое хранит элементы **в отсортированном порядке** и **без дубликатов**.

```java
TreeSet<Integer> ts = new TreeSet<>();
ts.add(5);    // → {5}
ts.add(2);    // → {2, 5}       — автоматически по порядку!
ts.add(8);    // → {2, 5, 8}
ts.add(2);    // → {2, 5, 8}    — дубликат НЕ добавился
```

Внутри `TreeSet` — **красно-чёрное дерево** (самобалансирующееся бинарное дерево поиска). Поэтому:
- добавление — O(log n)
- поиск — O(log n)
- элементы всегда отсортированы

**Важно:** чтобы класть свои объекты в `TreeSet`, они должны быть **сравнимыми** — реализовать `Comparable` (или передать `Comparator` в конструктор TreeSet).

## Часть 3. Класс Cex (ЦЕХ) — разбор по строчкам

```java
public class Cex implements Comparable<Cex> {
```
`implements Comparable<Cex>` — говорим Java: «объекты Cex можно сравнивать друг с другом». Без этого ни `TreeSet`, ни `Collections.sort()` не будут работать.

```java
    private final String name;          // название цеха: "Сборочный"
    private final String chief;         // начальник: "Иванов"
    private final int workersCount;     // количество работающих: 120
```
`private` — поля доступны только внутри класса.
`final` — после создания объекта изменить нельзя (immutable).

```java
    public Cex(String name, String chief, int workersCount) {
        this.name = name;
        this.chief = chief;
        this.workersCount = workersCount;
    }
```
**Конструктор** — вызывается при `new Cex("Сборочный", "Иванов", 120)`. Принимает три параметра и сохраняет их в поля.

```java
    public String getName()         { return name; }
    public String getChief()        { return chief; }
    public int    getWorkersCount() { return workersCount; }
```
**Геттеры** — поля `private`, снаружи читаем только через геттеры.

```java
    @Override
    public int compareTo(Cex o) {
        int c = Integer.compare(this.workersCount, o.workersCount);
        if (c != 0) return c;
        return this.name.compareTo(o.name);
    }
```
**compareTo** — сердце сравнения. Возвращает:
- **отрицательное число** → `this` меньше `o`
- **0** → равны
- **положительное** → `this` больше `o`

Логика: сначала сравниваем по количеству работающих. Если равно — по имени (алфавитный порядок). Это называется **естественный порядок**.

```java
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Cex)) return false;
        Cex cex = (Cex) o;
        return workersCount == cex.workersCount &&
               Objects.equals(name, cex.name) &&
               Objects.equals(chief, cex.chief);
    }
```
**equals** — проверяет, одинаковые ли два объекта **по содержимому**. Без переопределения Java сравнивала бы **по ссылке** (два объекта с одинаковыми полями считались бы разными).

```java
    @Override
    public int hashCode() {
        return Objects.hash(name, chief, workersCount);
    }
```
**hashCode** — числовой «отпечаток» объекта. Правило Java: если `a.equals(b) == true`, то `a.hashCode() == b.hashCode()`. Нужно для `HashMap`/`HashSet`.

```java
    @Override
    public String toString() {
        return String.format("Цех{имя='%s', начальник='%s', работающих=%d}",
                name, chief, workersCount);
    }
```
**toString** — что выведется при `System.out.println(cex)`. Без него было бы нечитаемое `by.bsuir.lab2.Cex@3f2a1c`.

## Часть 4. Программа №1 — Vector<Character>

Что делает: показывает базовые операции с вектором символов.

### Шаг за шагом:

```
Шаг 1: Создаём Vector, добавляем буквы a,b,c,d,e,f,g,h
        v1 = [a, b, c, d, e, f, g, h]

Шаг 2: Печатаем → [a, b, c, d, e, f, g, h]

Шаг 3: v1.remove(2)   — удаляем элемент №2 (это 'c', нумерация с нуля!)
        v1.set(0, 'X') — на место 'a' ставим 'X'
        v1.set(1, 'Y') — на место 'b' ставим 'Y'
        v1 = [X, Y, d, e, f, g, h]

Шаг 4: Обходим через Iterator:
        Iterator<Character> it = v1.iterator();
        while (it.hasNext()) {
            System.out.print(it.next() + " ");
        }
        Выведет: X Y d e f g h

Шаг 5: Создаём второй вектор v2 = [1, 2, 3, 4]

Шаг 6: Из v1 удаляем 2 элемента после индекса 1:
        удаляем v1[2]='d', потом v1[2]='e' (индексы сдвинулись!)
        v1 = [X, Y, f, g, h]
        Потом v1.addAll(v2) — приклеиваем все из v2
        v1 = [X, Y, f, g, h, 1, 2, 3, 4]

Шаг 7: Печатаем v1 и v2 (v2 не изменился: [1, 2, 3, 4])
```

### Что такое Iterator и зачем он нужен?

**Iterator** — объект для последовательного обхода коллекции. У него 3 метода:
- `hasNext()` — есть ли ещё элемент?
- `next()` — дай следующий элемент
- `remove()` — удали текущий элемент (единственный безопасный способ удалять при обходе)

```java
Iterator<Character> it = v1.iterator();  // получаем итератор
while (it.hasNext()) {                    // пока есть элементы
    Character c = it.next();              // берём следующий
    System.out.print(c);                  // что-то делаем
}
```

Зачем, если можно `for`? — Итератор **универсален** (работает с любой коллекцией, даже с теми, у которых нет индексов, как `TreeSet`). И только через `it.remove()` можно безопасно удалять при обходе.

## Часть 6. Программа №3 — алгоритмы (самое сложное)

Здесь 10 пунктов. Разбираю каждый.

### Пункт 1-3: Создание, сортировка, просмотр

```java
Vector<Cex> v = new Vector<>();
v.add(new Cex("Сборочный",  "Иванов",  120));
v.add(new Cex("Литейный",   "Петров",   80));
v.add(new Cex("Кузнечный",  "Сидоров", 150));
v.add(new Cex("Малярный",   "Жуков",    40));
v.add(new Cex("Покрасочный","Орлов",    65));
v.add(new Cex("Сварочный",  "Новиков",  95));

Collections.sort(v);  // сортировка по естественному порядку (compareTo)
```

После сортировки (по `workersCount`):
```
40  — Малярный
65  — Покрасочный
80  — Литейный
95  — Сварочный
120 — Сборочный
150 — Кузнечный
```

`Collections.sort(v)` использует `compareTo` класса `Cex` — сравнивает по `workersCount`.

### Пункт 4: Бинарный поиск

```java
Cex pattern = new Cex("Сварочный", "Новиков", 95);
int idx = Collections.binarySearch(v, pattern);  // → 3
```

**Бинарный поиск** — алгоритм поиска в **отсортированном** списке:
1. Смотрим на середину списка.
2. Если нашли — готово.
3. Если искомое меньше — ищем в левой половине.
4. Если больше — в правой.
5. Повторяем.

Сложность: **O(log n)** — для 1 000 000 элементов нужно всего ~20 сравнений!

Почему именно бинарный поиск, а не `indexOf`? — Потому что список уже **отсортирован** (пункт 2), и бинарный поиск это использует. `indexOf` не знает про порядок и проверяет все элементы подряд (O(n)).

### Пункт 5: Перемещение в TreeSet

```java
TreeSet<Cex> ts = new TreeSet<>();      // пустой TreeSet
Iterator<Cex> it = v.iterator();
while (it.hasNext()) {
    Cex c = it.next();
    if (c.getWorkersCount() >= 90) {    // условие: >= 90 работающих
        ts.add(c);                       // добавляем в TreeSet
        it.remove();                     // удаляем из Vector
    }
}
```

Результат:
- В `TreeSet` переехали: Сварочный(95), Сборочный(120), Кузнечный(150)
- В `Vector` остались: Малярный(40), Покрасочный(65), Литейный(80)

**ВАЖНО:** почему `it.remove()`, а не `v.remove(c)`?

```java
// ТАК НЕЛЬЗЯ — будет ConcurrentModificationException:
for (Cex c : v) {
    if (c.getWorkersCount() >= 90) {
        v.remove(c);    // ОШИБКА! Изменяем коллекцию во время обхода
    }
}

// ТАК МОЖНО — итератор знает, что мы удаляем:
Iterator<Cex> it = v.iterator();
while (it.hasNext()) {
    Cex c = it.next();
    if (c.getWorkersCount() >= 90) {
        it.remove();    // ОК — итератор сам обновляет внутреннее состояние
    }
}
```

### Пункт 7: Сортировка по убыванию

**Vector** — просто:
```java
v.sort(Comparator.reverseOrder());
// Было: 40, 65, 80  →  Стало: 80, 65, 40
```

**TreeSet** — **нельзя пересортировать**. Порядок задаётся **один раз** при создании. Поэтому создаём **новый** TreeSet с обратным компаратором:
```java
TreeSet<Cex> tsDesc = new TreeSet<>(Comparator.reverseOrder());
tsDesc.addAll(ts);    // копируем все элементы
ts = tsDesc;
// Было: 95, 120, 150  →  Стало: 150, 120, 95
```

### Пункт 9: Слияние

```java
List<Cex> merged = new ArrayList<>();
merged.addAll(v);     // все из Vector
merged.addAll(ts);    // все из TreeSet
merged.sort(Comparator.reverseOrder());
```

Тип третьего контейнера выбираем сами — `ArrayList` подходит, потому что он гибкий (есть индексы, можно сортировать, допускает дубликаты).

## Часть 7. Полный список вопросов препода и ответы

### Про Vector
**В: Чем Vector отличается от ArrayList?**
О: Все методы Vector — `synchronized` (потокобезопасные). ArrayList — нет, но работает быстрее в однопоточной программе. Мы используем Vector, потому что так указано в варианте.

**В: Что значит `synchronized`?**
О: Если два потока одновременно вызовут `v.add()`, Vector гарантирует, что они не испортят данные друг друга. ArrayList — не гарантирует.

### Про TreeSet
**В: Чем TreeSet отличается от HashSet?**
О: TreeSet хранит в отсортированном порядке (внутри — красно-чёрное дерево, O(log n)). HashSet — в произвольном порядке (хэш-таблица, O(1) на операцию, но порядок непредсказуем).

**В: Почему TreeSet не допускает дубликатов?**
О: Потому что это Set (множество). Если `compareTo` двух объектов возвращает 0, TreeSet считает их одинаковыми и не добавляет второй.

**В: Можно ли положить null в TreeSet?**
О: Нет (начиная с Java 7). `compareTo(null)` выбросит `NullPointerException`.

### Про Comparable
**В: Зачем Cex реализует Comparable?**
О: Чтобы объекты можно было сравнивать. Это нужно для: `Collections.sort()`, `Collections.binarySearch()`, `TreeSet` (который хранит элементы по порядку).

**В: Что будет, если не реализовать Comparable и положить в TreeSet?**
О: `ClassCastException` при первом `add()` — TreeSet не знает, как сравнивать.

**В: Можно ли обойтись без Comparable?**
О: Да, передав `Comparator` в конструктор TreeSet: `new TreeSet<>(Comparator.comparingInt(Cex::getWorkersCount))`. Но Comparable удобнее — один раз написал, везде работает.

### Про equals/hashCode
**В: Зачем переопределять equals?**
О: По умолчанию `equals` сравнивает ссылки (`==`). Два объекта с одинаковыми полями будут «не равны». Переопределяем, чтобы сравнивать по содержимому.

**В: Зачем переопределять hashCode вместе с equals?**
О: Контракт Java: если `a.equals(b)`, то `a.hashCode() == b.hashCode()`. Если нарушить — `HashMap`/`HashSet` будут работать неправильно (объект можно положить, но не найти).

**В: Контракт: equals=true → hashCode одинаков. А наоборот?**
О: Нет. Одинаковый hashCode не гарантирует equals=true. Это называется **коллизия** — нормальная ситуация для хэш-таблицы.

### Про Iterator
**В: Почему нельзя удалять из коллекции в for-each?**
О: `for-each` внутри использует итератор. Если изменить коллекцию напрямую (не через итератор), итератор обнаружит несоответствие и выбросит `ConcurrentModificationException`. Это механизм **fail-fast**.

**В: А через обычный for с индексом можно удалять?**
О: Технически да, но осторожно: после удаления индексы сдвигаются. Нужно уменьшать `i` после удаления или обходить с конца.

### Про сортировку
**В: Как работает Collections.sort()?**
О: Использует **TimSort** — гибрид сортировки вставками и слиянием. Сложность O(n log n). Требует, чтобы элементы реализовали `Comparable` (или передан `Comparator`).

**В: Что такое Comparator.reverseOrder()?**
О: Возвращает компаратор, который сравнивает в обратном порядке. Если естественный порядок: 40, 65, 80 → с `reverseOrder`: 80, 65, 40.

### Про autoboxing
**В: Что такое autoboxing?**
О: Автоматическое преобразование примитива в обёртку: `char 'a'` → `Character.valueOf('a')`. Обратное — **unboxing**: `Character` → `char`. Делает Java сам, программист не пишет явно.

**В: Есть ли overhead (затраты) у autoboxing?**
О: Да, создаётся объект в куче (heap). Для критичной производительности лучше использовать примитивные массивы. Для учебной задачи — несущественно.
