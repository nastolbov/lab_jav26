## Лабораторная работа №2 — Коллекции

### Условие (вариант 10)
- Контейнер тип 1 — **`Vector`**.
- Контейнер тип 2 — **`TreeSet`**.
- Встроенный тип данных — **`char`**.
- Пользовательский класс — **ЦЕХ**: `имя : String`, `начальник : String`,
  `количество работающих : int`.

Требуется написать **три программы**:

1. контейнер встроенного типа;
2. то же, но для пользовательского типа;
3. демонстрация алгоритмов работы с коллекциями.

### Структура проекта

```
lab2/
├── pom.xml
└── src/main/java/by/bsuir/lab2/
    ├── Main.java       — точка входа, вызывает Program1, Program2, Program3
    ├── Cex.java        — пользовательский класс «Цех», реализует Comparable
    ├── Program1.java   — Vector<Character> + операции из задания
    ├── Program2.java   — Vector<Cex>       + те же операции
    └── Program3.java   — Vector<Cex> ↔ TreeSet<Cex>: сортировка, поиск, слияние
```

### Что делает каждая программа

**Program1** (`Vector<Character>`):
1. создаёт `Vector<Character>` и заполняет символами `a..h`;
2. печатает контейнер;
3. удаляет один элемент (`v.remove(2)`) и заменяет два других (`v.set`);
4. печатает через `Iterator<Character>`;
5. создаёт второй `Vector<Character>` (`'1'..'4'`);
6. удаляет `n=2` элемента после индекса 1, добавляет все элементы второго
   контейнера через `addAll`;
7. печатает оба.

**Program2** (`Vector<Cex>`): то же самое, но с объектами `Cex`.

**Program3** — алгоритмы:
1. создаёт `Vector<Cex>`;
2. сортирует `Collections.sort(v)` — естественный порядок (`compareTo`
   по числу работающих);
3. печатает;
4. ищет элемент `Collections.binarySearch(v, pattern)` — подходящий алгоритм
   для отсортированного списка;
5. через `Iterator.remove()` перемещает элементы с `workersCount >= 90`
   из `Vector` в `TreeSet`;
6. печатает `TreeSet`;
7. сортирует оба контейнера по убыванию: `Vector` — `v.sort(reverseOrder())`,
   а `TreeSet` пересоздаётся с компаратором `reverseOrder()`, потому что
   у уже созданного `TreeSet` порядок изменить нельзя;
8. печатает оба;
9. сливает оба в третий контейнер `ArrayList<Cex>` и сортирует;
10. печатает третий контейнер.

### Возможные вопросы преподавателя

- **«Почему Cex реализует `Comparable`?»** — Потому что элементы кладутся
  в `TreeSet` (требует естественного порядка либо `Comparator`),
  и потому что `Collections.sort` и `binarySearch` используют этот порядок.
- **«Почему именно `binarySearch` — это “подходящий алгоритм” для пункта 4?»** —
  Контейнер уже отсортирован (пункт 2), а `binarySearch` работает за `O(log n)`.
  Если бы коллекция не была отсортирована, нужно было бы использовать
  линейный поиск (`indexOf`).
- **«Почему `TreeSet` нельзя “отсортировать по убыванию”, как Vector?»** —
  Порядок `TreeSet` определяется компаратором, заданным в конструкторе,
  и не меняется в процессе работы. Чтобы получить обратный порядок,
  создаётся новый `TreeSet` с `Comparator.reverseOrder()` и в него
  переносятся элементы.
- **«Что произойдёт, если два цеха имеют одинаковое число работающих
  и одинаковое имя?»** — `compareTo` вернёт 0, и `TreeSet` сочтёт их
  одинаковыми (не добавит дубликат). В `Vector` они оба останутся.
- **«Почему именно `Iterator.remove()`, а не `for (Cex c : v)
  v.remove(c)`?»** — Изменение коллекции во время `for-each` вызовет
  `ConcurrentModificationException`. `Iterator.remove()` — единственный
  безопасный способ удалять элементы при обходе.
- **«В чём отличие `Vector` от `ArrayList`?»** — `Vector` синхронизирован
  (методы `synchronized`), `ArrayList` — нет. Поэтому `Vector` медленнее
  в однопоточной среде, но потокобезопасен.
- **«Что было бы, если бы `Cex` не переопределял `equals`/`hashCode`?»** —
  `TreeSet` опирается на `compareTo`, поэтому работал бы. Но `HashSet`,
  `HashMap` и сравнение объектов `equals(obj)` сломались бы.

## Лабораторная работа №3 — Абстрактные классы, интерфейсы, Vaadin

### Условие (вариант 10)
Создать абстрактный базовый класс **`Array`** с виртуальными методами
`sort` и `foreach`. Производные классы:

- **`Insert`** — сортировка вставками, `foreach` = возведение в квадрат;
- **`Selection`** — сортировка выбором, `foreach` = вычисление логарифма.

Изменить программу, использовав в качестве базового класса
**интерфейс**.

UI организовать через **Vaadin**.

### Структура проекта

```
lab3/
├── pom.xml
└── src/main/java/by/bsuir/lab3/
    ├── Application.java   — Spring Boot + Vaadin (Theme + AppShellConfigurator)
    ├── AbstractArray.java — абстрактный класс с abstract sort()/foreach()
    ├── Insert.java        — сортировка вставками + квадрат
    ├── Selection.java     — сортировка выбором + Math.log
    ├── IArray.java        — интерфейс (второй вариант базового типа)
    ├── InsertI.java       — реализация IArray (вставки + квадрат)
    ├── SelectionI.java    — реализация IArray (выбор + логарифм)
    └── MainView.java      — Vaadin-страница (@Route)
```

### Что делает программа

1. На странице есть `TextField` для ввода чисел через запятую и кнопка
   «Выполнить».
2. По нажатию строка парсится в `double[]`. Если пустая или неверная —
   выводится `Notification` с описанием ошибки.
3. Создаются два объекта **через указатель на абстрактный класс**:
   ```java
   AbstractArray ins = new Insert(arr);
   AbstractArray sel = new Selection(arr);
   ```
   Это и есть демонстрация полиморфизма из задания.
4. Для каждого вызываются `sort()` и `foreachDefault()` — стандартные для
   наследника функции (квадрат для `Insert`, логарифм для `Selection`).
   Результат отображается в `Paragraph`-ах на странице.
5. Дополнительно — те же действия через ссылки **на интерфейс `IArray`**:
   `IArray iIns = new InsertI(arr); IArray iSel = new SelectionI(arr);`
   Показывает, что программу можно переписать без абстрактного класса,
   используя только интерфейс.

### Алгоритмы сортировки

**Сортировка вставками** (`Insert.sort`):
```text
для i от 1 до n−1:
    key = a[i]; j = i−1
    пока j ≥ 0 и a[j] > key:
        a[j+1] = a[j]; j--
    a[j+1] = key
```
Сложность: O(n²) в среднем и в худшем случае, O(n) в лучшем.

**Сортировка выбором** (`Selection.sort`):
```text
для i от 0 до n−2:
    min = i
    для j от i+1 до n−1:
        если a[j] < a[min]: min = j
    обмен a[i] и a[min]
```
Сложность: O(n²) всегда.

### Возможные вопросы преподавателя

- **«Чем абстрактный класс отличается от интерфейса?»** — Абстрактный класс
  может содержать поля, конструкторы, реализованные методы и хранит
  состояние (`protected double[] data`). Интерфейс описывает только
  контракт (с Java 8 — ещё `default`-методы), и класс может реализовать
  любое количество интерфейсов, но наследовать только один класс.
- **«Где здесь полиморфизм?»** — Переменная `AbstractArray ins`
  объявлена базовым типом, но хранит объект конкретного наследника
  `Insert`. При вызове `ins.sort()` JVM по таблице виртуальных методов
  выбирает реализацию из `Insert`. То же для `IArray`.
- **«Зачем `foreachDefault`?»** — Метод задания требует «поэлементная
  обработка» (квадрат / логарифм) — это разное поведение наследников.
  `foreach(DoubleUnaryOperator)` — общий, а `foreachDefault` фиксирует
  стандартную для класса операцию.
- **«Что делает `Math.log` от отрицательного / нулевого числа?»** —
  Возвращает `NaN` для `< 0` и `−Infinity` для `0`. В UI это видно
  как `NaN` / `-Infinity`. Это корректное поведение, документация
  `Math` его описывает.
- **«Почему именно `DoubleUnaryOperator`?»** — Это функциональный
  интерфейс из `java.util.function`, он работает с примитивом `double`
  без боксинга, что эффективнее, чем `Function<Double, Double>`.
- **«Зачем `clone()` в конструкторе?»** — Чтобы при дальнейших
  модификациях массива не менялся внешний массив, переданный в
  конструктор (защита от побочных эффектов).
- **«Почему Vaadin?»** — Это требование задания: «использовать web-интерфейс
  организованный с помощью Vaadin». Spring Boot + Vaadin — стандартный
  способ запуска Vaadin Flow приложения.

## Лабораторная работа №4 — Текстовые файлы и исключения

### Условие (вариант 10)
**Железная дорога.** Найти станцию с наибольшим количеством проходящих
через неё поездов, а также по запросу названий двух станций определить,
через какую из них проходит больше **пассажирских** поездов за сутки.

Формат строки файла:
```
Станция (строка из 20 символов) | пассажирских (int) | товарных (int)
```

Требования:
- интерфейс — Vaadin;
- меню — вертикальные `Tabs`;
- расположение — `HorizontalLayout`;
- заголовок страницы поместить в одну из ячеек `HorizontalLayout`;
- читать файл **двумя способами** — `FileReader/Scanner` и `Files.lines()`;
- если файл другого формата — сообщать об этом через **исключения**
  и отображать ошибку на странице.

### Структура проекта

```
lab4/
├── pom.xml
├── data/stations.txt              — пример входного файла (8 станций)
└── src/main/java/by/bsuir/lab4/
    ├── Application.java           — Spring Boot
    ├── Station.java               — POJO станции
    ├── StationFormatException.java— проверяемое исключение формата
    ├── StationParser.java         — парсинг одной строки
    ├── StationReader.java         — два способа чтения файла
    └── MainView.java              — Vaadin: меню, таблица, операции
```

### Что делает программа

1. На странице — поле «Имя файла», переключатель «Способ чтения»
   (`FileReader/BufferedReader` или `Files.lines()`) и кнопка «Загрузить».
2. После загрузки строки парсятся `StationParser.parse(...)`. Если
   формат неверный, бросается `StationFormatException`, оборачивается
   с указанием номера строки и выводится в красном `Paragraph`-е на
   странице.
3. Слева — `Tabs` (вертикальные) с тремя пунктами:
   - **«Таблица»** — `Grid<Station>` с колонками *Станция*,
     *Пассажирских*, *Товарных*, *Всего*;
   - **«Самая загруженная»** — кнопка «Найти»: проходит по списку и
     ищет станцию с максимальным `passenger + cargo`;
   - **«Сравнение»** — два `TextField` (станция A и станция B), кнопка
     «Сравнить»: ищет станции по имени, выводит, у какой больше
     пассажирских поездов и на сколько.
4. Заголовок «Лабораторная работа №4 …» помещён в левую колонку
   `HorizontalLayout` (`MainView extends HorizontalLayout`).

### Чтение файла двумя способами

`StationReader.readClassic(path)`:
```java
try (BufferedReader br = new BufferedReader(new FileReader(path.toFile()))) {
    String line;
    while ((line = br.readLine()) != null) { ... StationParser.parse(line) ... }
}
```

`StationReader.readFunctional(path)`:
```java
try (Stream<String> lines = Files.lines(path)) {
    return lines.filter(s -> !s.isBlank())
                .map(StationParser::parse)   // обёрнут в try/catch
                .collect(Collectors.toList());
}
```

### Возможные вопросы преподавателя

- **«Почему создано собственное исключение `StationFormatException`,
   а не `IOException`?»** — `IOException` описывает ошибки ввода/вывода,
  а здесь файл прочитан, но содержимое не соответствует формату.
  Это другая категория ошибок, поэтому собственное проверяемое исключение.
- **«Чем `Files.lines()` лучше/хуже, чем `BufferedReader`?»** —
  `Files.lines()` возвращает `Stream<String>`, можно использовать
  функциональные операции (`filter`, `map`, `collect`). `BufferedReader`
  — императивный стиль. По производительности они сравнимы,
  `Files.lines()` использует `BufferedReader` внутри.
- **«Почему `Stream` в `try-with-resources`?»** — `Files.lines()` держит
  открытым файловый дескриптор. Если не закрыть Stream, будет утечка
  ресурсов.
- **«Как пробрасываются проверяемые исключения через лямбды?»** — В лямбду
  нельзя бросить проверяемое исключение, поэтому в `map(...)`
  `StationFormatException` оборачивается в `RuntimeException`, а после
  `collect` — снова разворачивается и кидается наружу.
- **«Что вернётся, если в файле строка из 22 символов с правильным
  именем, но с неверными числами?»** — В `StationParser` сначала
  пробуем фиксированную ширину (20 символов имени + 2 числа), если
  числа не распарсились — переходим к гибкому разбору. Если и там
  не получилось — `StationFormatException` с описанием.
- **«Почему имя станции — 20 символов?»** — Это требование варианта.
  В парсере поддерживаются оба варианта: фиксированная ширина и
  гибкий разбор по пробелам, чтобы можно было использовать любой
  тестовый файл.
- **«Зачем `Tabs.Orientation.VERTICAL`?»** — Задание явно требует
  «организовать вертикальное меню» — это стандартный способ в Vaadin.
- **«Где обработка пустого файла?»** — Если файл пуст, `data` будет
  пустым списком, операции «Найти» и «Сравнить» выведут
  «Сначала загрузите файл».

## Лабораторная работа №5 — Бинарные файлы

### Условие (вариант 10)
Написать приложение, позволяющее **создавать** целочисленный бинарный
файл, **выводить** его и **обрабатывать**. Количество чисел в файле —
случайное из диапазона `1..n` (`n` вводится). Имена файлов запрашиваются
**через диалоговые окна**. Программа должна реагировать на любые
введённые данные.

Задание варианта 10:
> Выяснить, какие из чисел файла состоят только из нечётных цифр.
> Числа вывести в порядке возрастания, без повторений. Считывать
> элементы файла в массив или другую структуру нельзя, но для других
> целей её создать можно. Компоненты файла задаются случайно из диапазона
> `a..b`. Считать, что число элементов файла не ограничено конкретным
> числом. **При обработке файл можно прочитать только один раз.**

### Структура проекта

```
lab5/
├── pom.xml
└── src/main/java/by/bsuir/lab5/
    ├── Application.java       — Spring Boot
    ├── BinaryFileService.java — создание / чтение / обработка файла
    └── MainView.java          — Vaadin-страница и диалоги
```

### Что делает программа

1. Кнопка **«Создать файл…»** открывает `Dialog`, в котором нужно
   указать имя файла, `n` (макс. количество чисел), `a` и `b`
   (диапазон значений). После «Создать» сервис создаёт файл:
   - случайное `size = 1 + rnd.nextInt(n)` элементов;
   - каждое число `a + rnd.nextInt(b−a+1)`;
   - запись через `DataOutputStream.writeInt`.
2. Кнопка **«Выбрать файл…»** открывает `Dialog` для ввода имени
   существующего файла. После открытия его содержимое выводится
   в read-only `TextArea`.
3. Кнопка **«Обработать»** запускает поиск чисел из нечётных цифр.
   Алгоритм:
   - открывается `DataInputStream` с файлом;
   - в цикле читается по одному `int`-у, до `EOFException`;
   - **в массив элементы не складываются**;
   - проверяется `allDigitsOdd(v)`: число берётся по модулю,
     каждый раз `% 10` и `> 0`, если хоть одна цифра чётная —
     возврат `false`;
   - подходящие числа добавляются в `TreeSet<Integer>` —
     это автоматически даёт **сортировку по возрастанию** и
     **отсутствие повторов**;
   - результат печатается в `TextArea`.
4. Любые ошибки (неправильный путь, размер файла не кратен 4,
   отсутствие файла) перехватываются и показываются `Notification`-ом.

### Возможные вопросы преподавателя

- **«Почему `DataInputStream.readInt`, а не `FileChannel`/`ByteBuffer`?»**
  — Задание про учебную работу с бинарным файлом, `DataInputStream`
  — самый прямой способ читать `int`-ы из потока.
- **«Как Вы определяете конец файла?»** — Ловлю `EOFException`,
  бросаемое `readInt()`. Это стандартный приём для `DataInputStream`.
- **«Почему `TreeSet`, а не `ArrayList` + `Collections.sort` +
   удаление дубликатов?»** — `TreeSet` сразу сохраняет порядок
  и устраняет дубликаты за `O(log n)` на вставку, что компактнее.
  И при этом мы не нарушаем требование «не считывать элементы
  файла в массив или структуру»: в структуру (`TreeSet`) попадают
  не **элементы файла**, а **результаты обработки** — то, что
  «можно создавать структуру для других целей».
- **«Почему `Math.abs((long)n)`?»** — Если `n == Integer.MIN_VALUE`,
  то `Math.abs((int)n)` тоже будет `MIN_VALUE` (целочисленное
  переполнение). Приведение к `long` решает проблему.
- **«Цифра 0 — нечётная или чётная?»** — Чётная. Поэтому если число
  содержит ноль (например, 30), оно отбрасывается. И отдельно: если
  само число `0`, мы возвращаем `false`.
- **«Что произойдёт, если файл не кратен 4 байтам?»** — При попытке
  прочитать содержимое через `readAll` бросается `IOException`
  с пояснением «Размер файла не кратен 4 — это не бинарный int-файл».
  Пользователь увидит сообщение в `Notification`.
- **«Файл может быть прочитан только один раз — Вы это соблюдаете?»** —
  Да. В `oddDigitNumbers` есть единственный `try (DataInputStream...)`,
  и второй проход не делается. Метод `readAll` (для отображения
  содержимого) — это отдельная операция, которая тоже читает один
  раз; задание ограничивает только обработку.
- **«Почему `TextField` в диалоге, а не файловый выбор системы?»** —
  Vaadin Flow в браузере не имеет прямого доступа к файловой
  системе сервера. Стандартный путь — попросить пользователя
  ввести путь, что соответствует фразе задания «имя файла
  запрашивается с помощью диалоговых окон».
- **«Что если файл пустой?»** — Цикл `while(true){readInt}` сразу
  поймает `EOFException`, метод вернёт пустой `TreeSet`, в UI будет
  «Нет чисел, состоящих только из нечётных цифр».
